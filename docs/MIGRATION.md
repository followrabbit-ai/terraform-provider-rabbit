# Migrating an existing Rabbit setup to Terraform

If you already manage access in Rabbit through the UI and want to adopt
Terraform without rewriting every group by hand, follow this three-step
workflow. It uses Terraform's built-in [`import` blocks][1] together with the
companion `rabbit-tf-import` CLI shipped in this repo.

[1]: https://developer.hashicorp.com/terraform/language/import

## TL;DR

```sh
# 1. List your domain's existing groups as Terraform `import` blocks.
rabbit-tf-import --domain acme.com > imports.tf

# 2. Let Terraform fetch each group via the provider's Read and write the
#    matching resource HCL.
terraform plan -generate-config-out=generated.tf

# 3. Review generated.tf, fold it into your module, then apply — should be a
#    no-op because state already matches reality.
terraform apply
```

## Step-by-step

### 0. Install the helper

```sh
go install github.com/followrabbit-ai/terraform-provider-rabbit/cmd/rabbit-tf-import@latest
```

(Or `go build ./cmd/rabbit-tf-import` from a checkout — same flags either way.)

The CLI uses the same authentication chain as the provider:

| Flag | Env | Notes |
|---|---|---|
| `--endpoint` | `RABBIT_ENDPOINT` | Defaults to the prod endpoint. |
| `--audience` | `RABBIT_AUDIENCE` | Google OAuth2 client ID. Defaults to the prod client ID. |
| `--credentials` | `GOOGLE_CREDENTIALS` / `GOOGLE_APPLICATION_CREDENTIALS` | Service account JSON file or inline JSON. |
| `--impersonate-service-account` | `GOOGLE_IMPERSONATE_SERVICE_ACCOUNT` | Standard Google IAM impersonation. |

### 1. Generate import blocks

```sh
rabbit-tf-import --domain acme.com > imports.tf
```

`imports.tf` looks like:

```hcl
# Generated by rabbit-tf-import for domain "acme.com".
import {
  to = rabbit_group.platform_admins
  id = "acme.com/abcd-1234"
}

import {
  to = rabbit_group.viewers
  id = "acme.com/efgh-5678"
}
```

Resource names are slugged from each group's display name. Collisions get
`_2`, `_3` suffixes deterministically. Use `--resource-prefix` to namespace
when importing from multiple domains into one module.

By default only `rabbit_group` blocks are emitted, because `rabbit_group` is
already authoritative for its principal list. Pass `--include-members` to
also emit `rabbit_group_member` blocks per principal — only useful when you
deliberately want each principal managed by an additive resource (e.g. one
group split across several modules / teams).

### 2. Let Terraform write the HCL

Make sure your module already declares the provider:

```hcl
terraform {
  required_providers {
    rabbit = {
      source  = "followrabbit-ai/rabbit"
      version = "~> 0.1"
    }
  }
}

provider "rabbit" {
  domain_id = "acme.com"
}
```

Then:

```sh
terraform init
terraform plan -generate-config-out=generated.tf
```

Terraform calls the provider's `ImportState` and `Read` for each `import`
block and emits a corresponding `resource "rabbit_group" "..." { ... }`
stanza into `generated.tf`. `terraform plan` will report each as "to be
imported, no changes".

### 3. Polish and apply

Quick clean-ups worth doing before the first apply:

- Replace literal role IDs (`roles = ["roles/domain.viewer"]`) with
  `data.rabbit_role.viewer.id` references — better readability, fewer
  hard-coded magic strings.
- Move `generated.tf` content into the file layout you actually want
  (`groups.tf`, per-team files, etc.). Delete `generated.tf` afterwards.
- Delete `imports.tf` once the first apply succeeds. `import` blocks are
  single-use; leaving them in is harmless but noisy.

Then:

```sh
terraform apply
```

The plan should be empty — every resource was imported with its current
state, so the apply is a no-op.

## What if I want some groups out of Terraform's reach?

`rabbit-tf-import` only emits what `ListGroups` returns, so simply delete
the `import` blocks for groups you don't want to manage with Terraform
before running step 2. Terraform won't manage what it doesn't import.

## Re-running the helper

It's safe to re-run `rabbit-tf-import` later. Use it to:

- Bring newly UI-created groups under Terraform management.
- Diff against your current `imports.tf` to spot UI-only changes.

The output is stable and review-friendly (groups sorted by name, members
sorted by `(type, name)`).
